# 프로젝트 개요 (Project Overview)

> 이 문서는 2026년 1학기 캡스톤디자인 기간 동안 Notion에 기록된 회의록·요구사항 문서(SRS/PRD/SAD/USECASE)·발표자료를 근거로 작성되었습니다. **이 저장소(GitHub)에는 아직 실제 소스 코드가 커밋되어 있지 않으며**, 아래 내용은 모두 팀의 Notion 기록을 원본 자료로 삼아 정리한 것입니다. 자세한 근거는 각 항목에 괄호로 표기한 회의록 날짜를 참고하십시오.

## 1. 프로젝트를 시작한 이유 / 해결하려는 문제

PRD에 명시된 문제의식은 다음과 같습니다.

> "헬스에 대한 관심은 많지만 오래 지속하기 힘들다(동기부여 부족), 잘못된 자세로 인한 부상 위험이 있다." (PRD)

이에 대한 해결책으로 "자동 기록을 통한 동기부여와 루틴 추천", "자세를 텍스트뿐 아니라 시각자료로 직관적으로 설명"하는 방향을 제시했습니다. SRS에는 "본 문서는 영상 기반 운동 보조 시스템의 SW 요구사항을 정의한다. 시스템은 사용자의 운동 영상을 분석하여 운동 종목 인식, 반복 횟수 계산, 자동일지 생성, 자세 피드백 기능을 제공한다"라고 명시되어 있습니다 (SRS).

초기 아이디어 논의(3월 초) 단계에서는 "운동 앱(비디오 모션인식 + 횟수, LLM 루틴, 자세교정)"이 제안되었고, 이후 0312 교수 피드백에서 "자세를 누가 판단할 것이냐"라는 지적을 받아 "모범(전문가) 영상과 비교"하는 방향으로 구체화되었습니다 (아이디어 논의, 0312 피드백).

## 2. 대상 운동

스쿼트(`squat`) · 벤치프레스(`benchpress`) · 데드리프트(`deadlift`) 3대 운동입니다. 종목명 표기는 `squat` / `benchpress` / `deadlift`로 통일하며, 회의록 원문에는 `bechpress`, `beachpress`, `sqaut`와 같은 오타가 일부 존재했습니다(0428). `squant`라는 표기는 이번 조사에서 확인되지 않았습니다.

## 3. 최종 목표

USECASE 문서 기준으로 시스템은 다음 4가지 유스케이스로 구성됩니다 (USECASE).

1. **운동 영상 기록**: 촬영/업로드 → 종목 분류 → 반복 횟수 계산 → 기록 생성 → 조회
2. **자세 피드백**: 사용자 + 기준(전문가) 영상 업로드 → 관절 추출 → 영상 합성 → 피드백 제공
3. **운동 기록 수동 입력/수정** (옵션)
4. **운동 계획 관리** (옵션, LLM 기반 루틴 추천)

## 4. 지시된 7대 기능과 실제 요구사항 문서의 대응 관계

| 기능 | SRS/USECASE 대응 | 비고 |
|---|---|---|
| 운동 종목 인식 | FR-03, UC2 | 명시적으로 정의됨 |
| phase 분석 | 명시적 요구사항 없음 | 0409 이후 실험을 통해 부각된 개념 (아래 참고) |
| 운동 횟수 카운팅 | FR-04, UC3 | 명시적으로 정의됨 |
| 사용자-전문가 자세 비교 | FR-09, UC5~7 | 명시적으로 정의됨 |
| 운동 자세 오류 판단 | 명시적 요구사항 없음 | 0423 교수 피드백에서 "정상/비정상 분류" 개념으로 논의 |
| 시각적 자세 피드백 | FR-10, FR-11 | 명시적으로 정의됨 |
| 3개 운동 통합 | 명시적 요구사항 없음 | 0423 교수 피드백("모델 하나로 만들고 branch로 나눠라")에서 대두 |

즉 7개 기능 중 4개(종목인식/횟수카운팅/자세비교/시각피드백)는 SRS/PRD/USECASE에 공식 요구사항으로 기록되어 있고, 나머지 3개(phase 분석/오류판단/통합)는 학기 중 발표 과정에서 지도교수 피드백을 통해 점진적으로 구체화된 것으로 확인됩니다. 이는 "1학기 초기 계획"과 "1학기 진행 중 확장된 범위"를 구분하기 위해 기록해 둡니다.

## 5. 비기능 요구사항 (SRS 기준)

- 20FPS 이상 처리
- 종목 인식 정확도 80% 이상
- 횟수 계산 정확도 90% 이상
- 2초 이내 응답
- 모바일 대응
- 10분 이상 연속 처리 안정성

이 수치들은 SRS에 목표치로 기술된 것이며, 1학기 말 기준 실제 달성 여부는 [`experiments_and_results.md`](./experiments_and_results.md)를 참고하십시오(예: 종목 인식 실측 accuracy 92.31%는 목표 초과 달성했지만, 실제 영상 재검증에서 일부 오분류가 발생했습니다).

## 6. 사용 기술 스택

Notion에 명시적으로 등장한 기술만 기록합니다.

- **포즈 추정**: MediaPipe Pose(33 keypoints, 주 트랙), YOLOv8n-pose(COCO 17 keypoints, 비교 실험 트랙)
- **종목 인식 / 시계열 모델**: ST-GCN, TCN, CTR-GCN, Shift-GCN, 2s-AGCN (검토·실험됨. 실사용 채택 여부는 문서별로 다름 — [`experiments_and_results.md`](./experiments_and_results.md) 참고)
- **바벨 검출(실험)**: YOLO-World (실험 완료, 성능 미달로 미채택 — [`limitations.md`](./limitations.md) 참고)
- **백엔드/프론트엔드(SAD 기준 계획)**: FastAPI, React, MySQL — 단, 이 항목들이 실제로 구현되었다는 근거는 확인되지 않았으며 SAD 문서의 계획 단계 기술로만 확인됩니다.
- **LLM API 비교 검토**: OpenAI, Google Gemini, Anthropic Claude, xAI Grok, Cohere, DeepSeek (코칭 문장·운동일지 자동 생성 용도로 비용 비교만 수행 — [`../README.md`](../README.md) 2학기 계획 참고)
- Python 기반 개발로 추정되나(파일 확장자 `.py`, `.ipynb`가 다수 언급), `requirements.txt` 작성 근거가 되는 완전한 import 목록은 실제 코드가 저장소에 없어 확인할 수 없었습니다. 자세한 내용은 저장소 루트의 [`requirements.txt`](../requirements.txt) 상단 주석을 참고하십시오.

## 7. 팀원 및 역할

Notion "신상" 데이터베이스의 팀원 개인 페이지(고준영, 김민식, 강완석, 유영진)에는 이름/이메일/전화번호/학번 속성만 있고 역할 설명은 비어 있었습니다. 개인정보(전화번호, 학번, 이메일)는 공개 저장소 정책상 이 문서에 옮기지 않았습니다. 대신 회의록에서 확인된 담당 업무만 기록합니다.

| 이름 | 회의록에서 확인된 담당 업무 | 근거 |
|---|---|---|
| 강완석 | 스쿼트 자세 피드백 담당 | 0430 |
| 유영진 | 벤치프레스 자세 피드백 담당 | 0430 |
| 김민식 | 자세 지표 계산 공식(각도/거리) 정리, GitHub 저장소 커밋 계정(`minsik0129`) | 0430, GitHub 커밋 이력 |
| 고준영 | 초기 프로젝트 아이디어(모션인식+LLM 루틴 앱) 제안 | 아이디어 논의 |

데드리프트 담당자는 회의록에서 명시적으로 특정되지 않았습니다(언급 없음). 2학기에 역할 배정을 명확히 하는 것을 권장합니다.
